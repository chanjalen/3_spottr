SECTION 2: VIEWS
================================================================================

VIEW 1: HttpResponse (Manual)
-----------------------------
Name: gym_list_manual
URL: /gyms/manual/
File: gyms/views.py

Purpose: Demonstrates the manual approach to rendering templates using
loader.get_template() and wrapping the result in HttpResponse.

When useful: When you need fine-grained control over the response, such as
setting custom headers, or when working with non-HTML content types.


VIEW 2: render() Shortcut
-------------------------
Name: gym_list_render
URL: /gyms/render/
File: gyms/views.py

Purpose: Demonstrates the shortcut approach using Django's render() function
which combines template loading and HttpResponse in one step.

When useful: For most standard views - it's cleaner, more readable, and
handles the common case of rendering a template with context data.


VIEW 3: Base Class-Based View
-----------------------------
Name: GymListView
URL: /gyms/cbv/
File: gyms/views.py

Purpose: Demonstrates inheriting from django.views.View and manually
implementing the get() method to handle requests.

When useful: When you need more control than generic views provide, or when
your view logic doesn't fit the patterns of generic views (e.g., complex
multi-model queries, custom business logic).


VIEW 4: Generic Class-Based Views
---------------------------------
Names: GymGenericListView, GymDetailView
URLs: /gyms/generic/, /gyms/<int:pk>/
File: gyms/views.py

Purpose: Demonstrates using Django's built-in generic views (ListView,
DetailView) which handle common patterns automatically.

When useful: For standard CRUD operations - less code, follows Django
conventions, and provides built-in functionality like pagination.


================================================================================
EXPERIENCE NOTES (Bonus)
================================================================================

HttpResponse vs render():
- render() is preferred for 99% of cases - it's cleaner and does the same thing
- HttpResponse is useful when you need to set custom headers or return non-HTML

Base CBV vs Generic CBV:
- Generic CBVs (ListView, DetailView) are great for standard CRUD operations
- Base CBV is better when you have complex logic that doesn't fit the generic pattern
- We found Generic CBVs saved a lot of boilerplate code for simple list/detail views

QUESTION 1: The Size vs. Quality Trade-off
================================================================================

Based on our 15-model benchmark in llm-test/ai_prototype.ipynb, the "sweet spot"
for Spottr's workout summary use case is the Small tier (1B–3B parameters),
specifically TinyLlama-1.1B-Chat-v1.0.

BENCHMARK RESULTS SUMMARY
--------------------------
| Tier         | Example Model               | Params  | Local Gen (s) | Output Quality (1-5) |
|--------------|----------------------------|---------|---------------|----------------------|
| Ultra-Light  | distilgpt2                 | 82 M    | ~0.5s         | 1 — incoherent       |
| Ultra-Light  | gpt2                       | 124 M   | ~0.6s         | 1 — repetitive       |
| Ultra-Light  | Qwen2-0.5B-Instruct        | 494 M   | ~1.2s         | 2 — on-topic but choppy |
| Small        | TinyLlama-1.1B-Chat        | 1.1 B   | ~2.5s         | 3.5 — good structure |
| Small        | stablelm-2-zephyr-1.6B     | 1.6 B   | ~3.8s         | 3 — acceptable       |
| Small        | phi-2                      | 2.7 B   | ~6s           | 4 — solid quality    |
| Medium       | Phi-3-mini-4k-instruct     | 3.8 B   | ~12s          | 4.5 — excellent      |
| Medium       | Llama-3.2-3B-Instruct      | 3.2 B   | ~10s          | 4.5 — excellent      |
| Medium       | Mistral-7B-Instruct-v0.1   | 7.2 B   | ~35s          | 5 — near-perfect     |
| Large        | Qwen2-7B-Instruct (Groq)   | 7.6 B   | ~0.8s*        | 5 — near-perfect     |
| Large        | gemma-7b-it (Groq)         | 7 B     | ~0.7s*        | 4.5 — excellent      |
| Large        | Llama-2-13B (Groq)         | 13 B    | ~0.9s*        | 5 — near-perfect     |
| XLarge       | Mixtral-8x7B (Groq)        | 46.7 B  | ~1.1s*        | 5 — best             |
| XLarge       | Llama-2-70B (Groq)         | 70 B    | ~1.3s*        | 5 — best             |
| XLarge       | Qwen2-72B (Groq)           | 72 B    | ~1.2s*        | 5 — best             |

* Groq cloud API — latency does not reflect local compute cost

Was a massive model significantly better?
Yes, but with rapidly diminishing returns for our specific task. Workout summaries are
short, structured, and formulaic — a 70B model is not meaningfully better than a 7B model
for this task. The biggest jump was Ultra-Light → Small (unusable to usable), and a
secondary jump at Small → Medium (good to great). XLarge vs Large showed almost no
perceptible difference for 2–3 sentence workout summaries.

ESTIMATED SERVER COMPUTE COST TO RUN LOCALLY
---------------------------------------------
Assumptions: t3.xlarge AWS equivalent (4 vCPU, 16 GB RAM, no GPU) = ~$0.166/hr

| Model Size  | RAM Needed | GPU Needed | Cost/1K requests (CPU) |
|-------------|------------|------------|------------------------|
| <1B         | ~2 GB      | No         | ~$0.002                |
| 1B–3B       | ~3–6 GB    | No (slow)  | ~$0.01                 |
| 3B–7B       | ~8–14 GB   | Recommended| ~$0.05–0.15            |
| 7B–13B      | ~16–28 GB  | Required   | ~$0.30–0.80            |
| >13B        | ~40–150 GB | Required   | ~$2.00–10.00           |

Verdict: Small (1B–3B) is the sweet spot — cheap enough to self-host, fast enough for
user-facing features, and quality is acceptable for short structured outputs.

QUESTION 2: Cost & Scaling — 10,000 Daily Users
================================================================================

SCENARIO
--------
10,000 daily users, each triggering 1 AI summary per workout session.
Assume average 1 workout logged per user per day = 10,000 API calls/day.

LOCAL DJANGO (TinyLlama-1.1B on CPU server)
--------------------------------------------
- Average inference time: ~2.5s per request
- Peak concurrency: ~500 requests at once (assuming spread over day)
- Servers needed: ceil(500 * 2.5 / 60) ≈ 21 workers
- Recommended instance: c6i.4xlarge (16 vCPU, 32 GB RAM) = $0.68/hr
- Monthly cost: ~$490/month (1 instance running 24/7 for 10K users)
- Plus: auto-scaling overhead, storage, DevOps time

GROQ API (llama3-8b-8192, free tier capped at 14,400 req/day)
---------------------------------------------------------------
- Free tier covers 10,000 req/day with room to spare
- If scaling to 100K users: paid tier at approximately $0.05 per 1M tokens
  - Average tokens per request: ~300 (prompt + response)
  - 100K users * 300 tokens = 30M tokens/day = ~$1.50/day = ~$45/month
- No infrastructure to manage

COMPARISON
----------
| Option           | 10K users/day cost | 100K users/day cost | Latency | Offline? |
|------------------|--------------------|---------------------|---------|----------|
| Local TinyLlama  | ~$490/month        | ~$4,900/month       | 2–4s    | Yes      |
| Groq API         | $0 (free tier)     | ~$45/month          | <1s     | No       |

Decision: At 10,000 users we would move entirely to the Groq API. The cost and
infrastructure savings are overwhelming, and Groq's free tier comfortably handles the load.
We keep the local model only as an offline fallback.

QUESTION 3: Hybrid Potential
================================================================================

PROPOSED HYBRID PIPELINE
-------------------------

Step 1 — Local model (TinyLlama) as a PRE-FILTER:
  - Run the user's input through TinyLlama locally to detect:
    a) Off-topic/non-workout content (prompt injection attempts)
    b) Inputs that are already so simple a small model can handle them well
       (e.g., "30 min walk" — no need for a 70B model)
  - TinyLlama classifies: "simple" or "complex" workout

Step 2 — Route based on classification:
  - Simple workout → TinyLlama generates the summary locally (free, fast)
  - Complex workout (many exercises, PRs, unusual movements) → Groq API

ESTIMATED SAVINGS
-----------------
If 60% of workouts are "simple" (e.g., single-exercise cardio):
  - 60% handled locally: $0 cost
  - 40% sent to Groq: 40% of token cost
  - Net saving: ~60% reduction in API spend
  - Quality maintained: complex workouts still get the best model

This hybrid architecture mirrors production ML systems like Anthropic's model routing
(haiku for simple tasks, opus for complex) and minimises cost without sacrificing UX.

QUESTION 4: Pre-trained Model vs. Paid API — Deployment Thresholds
================================================================================

SERVER SPEC: 16-core CPU, 16-core GPU, 64 GB RAM, 1 TB SSD

At this spec, the server can comfortably host:
- Up to 13B parameter models in float16 (~26 GB VRAM equivalent)
- With quantisation (4-bit): up to ~30–40B models

WHEN TO USE THE PRE-TRAINED LOCAL MODEL
----------------------------------------
Condition 1: Request volume is HIGH and predictable
  - If serving >50,000 requests/day, the server becomes cheaper than API per-token pricing
  - Break-even vs Groq paid tier: ~$0.05/1M tokens; at 64 GB RAM server cost of ~$800/month
    → break-even at ~16B tokens/month → ~533M tokens/day → ~1.8M short requests/day

Condition 2: Data privacy is critical
  - If workout data contains sensitive health information that cannot leave the server
  - Healthcare compliance (HIPAA) may prohibit sending data to third-party APIs

Condition 3: Offline / edge deployment
  - Gym environments with unreliable internet
  - Embedded devices or kiosks

Condition 4: Latency requirements are very tight
  - If the application requires <100ms response (e.g., real-time form feedback)
  - Local GPU inference on 7B model: ~200–500ms vs Groq ~700ms round-trip

WHEN TO SWITCH TO THE PAID API
--------------------------------
Condition 1: Volume is LOW (<50,000 req/day)
  - The $800+/month server cost is not justified; API is cheaper per request

Condition 2: You need a model larger than 13B (full quality)
  - 64 GB RAM cannot run 70B models without aggressive quantisation that degrades quality
  - Groq runs Llama-3 70B at full quality for a fraction of the server cost

Condition 3: Rapid iteration / model upgrades
  - APIs automatically provide access to latest model versions
  - No re-deployment required when a better model is released

Condition 4: Burst traffic / unpredictable load
  - A fixed server cannot scale to 10x traffic without over-provisioning
  - APIs scale elastically with demand

SUMMARY TABLE
-------------
| Scenario                          | Recommendation        |
|-----------------------------------|-----------------------|
| >1M requests/day, GPU server      | Self-hosted local     |
| Privacy/compliance required       | Self-hosted local     |
| <50K requests/day                 | Paid API              |
| Need >13B model at full quality   | Paid API              |
| Burst/unpredictable traffic       | Paid API              |
| Tight latency + GPU available     | Self-hosted local     |
| Startup / MVP stage               | Paid API (Groq free)  |
